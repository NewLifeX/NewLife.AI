## 2. 项目结构与依赖

### 2.1 项目信息

- 项目名称：`NewLife.ChatAI`
- 项目类型：ASP.NET Core Web
- 目标框架：`net8.0`
- 必须引用：`NewLife.Cube.Core`
- 前后端同项目：前端静态资源放 `wwwroot/`，API 放 `Controllers/`

系统分为两层：

| 层级 | 项目 | 职责 |
|------|------|------|
| 基础库 | `NewLife.AI` | 多协议模型适配（OpenAI/Anthropic/Gemini）、21+ 服务商支持、函数调用、图像生成等通用能力。不依赖数据库 |
| 应用层 | `NewLife.ChatAI` | 对话 Web 前端、会话管理、API 网关路由、MCP 工具管理、使用量统计等业务功能。依赖 XCode 数据库 |

## 3. 数据模型

### 3.1 核心实体表

以下表定义在 `NewLife.ChatAI/Entity/Model.xml` 中，使用 XCode 实体框架。详细字段定义见 Model.xml，此处仅列出表清单。

| # | 表名（实体名） | 中文名 | 说明 |
|---|---------------|--------|------|
| 1 | Conversation | 会话 | 一次完整的多轮对话上下文 |
| 2 | ChatMessage | 对话消息 | 会话中的单条发言，包括用户消息和 AI 回复 |
| 3 | MessageFeedback | 消息反馈 | 用户对 AI 回复的点赞或点踩 |
| 4 | SharedConversation | 共享会话 | 通过链接分享的对话快照 |
| 5 | ModelConfig | 模型配置 | 后端接入的大语言模型配置，含提供商、接口地址、能力标识等 |
| 6 | UserSetting | 用户设置 | 用户的个性化配置（语言、主题、默认模型等） |
| 7 | AppKey | 应用密钥 | API 网关访问凭证，支持按 Key 维度用量统计（需求 §7.5） |
| 8 | UsageRecord | 用量记录 | 每次 AI 调用的 Token 消耗，关联用户和 AppKey（需求 §10） |
| 9 | McpServerConfig | MCP服务配置 | MCP Server 列表及工具发现信息（需求 §9） |

> **注意**：不支持消息分支，ChatMessage 无 `ParentMessageId` 字段。编辑/重新生成直接替换当前消息。

### 3.2 附件存储

附件 **不** 使用独立的 `ChatAttachment` 表，而是复用 **魔方（NewLife.Cube）已有的 `Attachment` 表**。

魔方附件表关键字段：

| 字段 | 说明 |
|------|------|
| Id | Int64，雪花 ID |
| Category | 业务分类，填 `ChatAI` |
| Key | 业务主键，可存消息 ID |
| FileName | 原始文件名 |
| Extension | 扩展名 |
| Size | 文件大小 |
| ContentType | MIME 类型 |
| FilePath | 本地相对路径 |

使用方式：
- 上传附件时，`Category` 设为 `"ChatAI"`，`Key` 设为消息 ID（发送前为 `"temp_用户ID"`）。
- 消息的 `Attachments` 字段存储附件 ID 数组的 JSON，如 `[123456, 789012]`。
- 下载/预览通过魔方附件控制器，或封装为 `/api/attachments/{id}` 代理。

## 4. API 接口规范

### 4.1 统一错误响应格式

所有接口错误均返回以下 JSON 结构：

```json
{
  "code": "ERROR_CODE",
  "message": "人类可读的错误描述",
  "traceId": "链路追踪ID"
}
```

常用错误码：

| 错误码 | HTTP 状态码 | 说明 |
|--------|-----------|------|
| `CONVERSATION_NOT_FOUND` | 404 | 会话不存在 |
| `MESSAGE_NOT_FOUND` | 404 | 消息不存在 |
| `MODEL_UNAVAILABLE` | 503 | 模型服务不可用 |
| `CONTEXT_TOO_LONG` | 400 | 上下文超出模型限制 |
| `RATE_LIMITED` | 429 | 请求过于频繁 |
| `FILE_TOO_LARGE` | 413 | 附件超出大小限制 |
| `UNSUPPORTED_FILE_TYPE` | 415 | 不支持的文件类型 |
| `SHARE_EXPIRED` | 410 | 共享链接已过期 |
| `SHARE_NOT_FOUND` | 404 | 共享链接不存在 |
| `TOOL_CALL_FAILED` | 500 | 工具调用执行失败 |
| `MCP_SERVER_UNAVAILABLE` | 503 | MCP Server 不可达 |
| `IMAGE_GENERATION_FAILED` | 500 | 图像生成失败 |
| `INVALID_API_KEY` | 401 | AppKey 无效或已禁用 |

## 5. SSE 流式协议规范

### 5.1 协议概述

发送消息接口 `POST /api/conversations/{id}/messages` 以 **Server-Sent Events（SSE）** 格式返回响应。

HTTP 响应头：

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

每条 SSE 事件格式：

```
data: {JSON}\n\n
```

### 5.2 请求格式

```json
{
  "content": "用户输入的消息文本",
  "thinkingMode": 0,
  "attachmentIds": [123456, 789012]
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| content | String | 是 | 消息文本内容 |
| thinkingMode | Int32 | 否 | 0=自动(默认), 1=思考, 2=快速 |
| attachmentIds | Int64[] | 否 | 附件 ID 列表 |

### 5.3 SSE 事件类型

#### 5.3.1 消息头事件（message_start）

流开始时发送，包含消息的元信息：

```
data: {"type":"message_start","messageId":5001,"model":"qwen-max","thinkingMode":1}
```

| 字段 | 说明 |
|------|------|
| type | 固定 `"message_start"` |
| messageId | AI 回复的消息 ID |
| model | 实际使用的模型编码 |
| thinkingMode | 实际使用的思考模式 |

#### 5.3.2 思考过程增量事件（thinking_delta）

思考模式下，输出模型的推理过程：

```
data: {"type":"thinking_delta","content":"让我分析一下这个问题..."}
```

| 字段 | 说明 |
|------|------|
| type | 固定 `"thinking_delta"` |
| content | 思考内容的增量文本 |

#### 5.3.3 思考完成事件（thinking_done）

思考过程结束：

```

| 字段 | 说明 |
|------|------|
| type | 固定 `"thinking_done"` |
| thinkingTime | 思考耗时，毫秒 |

#### 5.3.4 内容增量事件（content_delta）

```

| 字段 | 说明 |
|------|------|
| type | 固定 `"content_delta"` |
| content | 回答内容的增量文本 |

#### 5.3.5 完成事件（message_done）

全部内容输出完毕：

```
data: {"type":"message_done","usage":{"promptTokens":150,"completionTokens":320,"totalTokens":470}}
```

| 字段 | 说明 |
|------|------|
| type | 固定 `"message_done"` |

| usage.promptTokens | 提示 Token 数 |
| usage.completionTokens | 回复 Token 数 |
| usage.totalTokens | 总 Token 数 |

#### 5.3.6 错误事件（error）

| 字段 | 说明 |
|------|------|
| type | 固定 `"error"` |
| code | 错误码 |

#### 5.3.7 工具调用开始事件（tool_call_start）（需求 §8.3）

#### 5.3.8 工具调用完成事件（tool_call_done）

### 5.4 完整 SSE 流示例

#### 快速模式（无思考过程）

```
data: {"type":"message_start","messageId":5001,"model":"qwen-max","thinkingMode":2}

data: {"type":"content_delta","content":"你好！"}

data: {"type":"content_delta","content":"我是AI助手，"}

data: {"type":"content_delta","content":"有什么可以帮你的？"}

data: {"type":"message_done","usage":{"promptTokens":10,"completionTokens":15,"totalTokens":25}}
```

#### 思考模式（含思考过程）

```
data: {"type":"message_start","messageId":5002,"model":"deepseek-r1","thinkingMode":1}

data: {"type":"thinking_delta","content":"用户问的是一个数学问题，"}

data: {"type":"thinking_delta","content":"我需要用二次方程求根公式..."}

data: {"type":"thinking_delta","content":"判别式 Δ = b²-4ac = ..."}

data: {"type":"thinking_done","thinkingTime":3200}

data: {"type":"content_delta","content":"根据求根公式，方程的解为：\n\n"}

data: {"type":"content_delta","content":"$$x = \\frac{-b \\pm \\sqrt{b^2-4ac}}{2a}$$"}

data: {"type":"message_done","usage":{"promptTokens":50,"completionTokens":120,"totalTokens":170}}
```

#### 错误中断

```
data: {"type":"message_start","messageId":5003,"model":"gpt-4o","thinkingMode":0}

data: {"type":"content_delta","content":"让我来回答"}

data: {"type":"error","code":"MODEL_UNAVAILABLE","message":"模型连接超时，请稍后重试"}
```

#### 交错思考 + 工具调用（需求 §6.3.1 / §8）

```
data: {"type":"message_start","messageId":5004,"model":"gpt-4o","thinkingMode":1}

data: {"type":"thinking_delta","content":"用户想知道北京天气，我需要调用天气工具..."}

data: {"type":"thinking_done","thinkingTime":800}

data: {"type":"tool_call_start","toolCallId":"call_001","name":"get_weather","arguments":"{\"city\":\"北京\"}"}

data: {"type":"tool_call_done","toolCallId":"call_001","result":"{\"temp\":25,\"weather\":\"晴\"}","success":true}

data: {"type":"thinking_delta","content":"天气数据已获取，北京25度晴天，我来组织回复..."}

data: {"type":"thinking_done","thinkingTime":500}

data: {"type":"content_delta","content":"北京今天天气晴好，气温 25°C，"}

data: {"type":"content_delta","content":"适合户外活动。"}

data: {"type":"message_done","usage":{"promptTokens":80,"completionTokens":60,"totalTokens":140}}
```

### 5.5 前端处理逻辑

1. 收到 `message_start`：创建消息气泡，显示模型图标和加载指示器。
2. 收到 `thinking_delta`：在折叠区域追加思考过程文本，显示"思考中..."标题。
3. 收到 `thinking_done`：关闭思考区域，标题改为"思考过程（耗时 X.X 秒）"。
4. 收到 `tool_call_start`：在对话流中插入工具调用块（ToolCallBlock），显示工具名称、参数和旋转图标。
5. 收到 `tool_call_done`：更新工具调用块状态为成功（✓），显示返回结果（可折叠）。
6. 收到 `tool_call_error`：更新工具调用块状态为失败（✗），显示错误原因。
7. 收到 `content_delta`：在正式回答区域追加内容，进行 Markdown 增量渲染。
8. 收到 `message_done`：移除加载指示器，显示操作按钮，可展示 Token 用量。
9. 收到 `error`：停止流，显示错误提示，已输出内容保留。
10. 连接断开且未收到 `message_done` / `error`：视为异常中断，保留已有内容并提示。

> **交错渲染**：步骤 2–6 可交替出现多次（think–tool–think 链路），前端按事件到达顺序依次渲染即可。

## 6. 系统配置

### 6.1 ChatSetting 配置类

继承 `Setting<ChatSetting>`，提供系统级配置。配置文件默认位于 `Config/ChatSetting.config`。

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| ShareExpireDays | Int32 | 30 | 分享有效期。共享链接有效天数，0 表示永不过期 |
| DefaultModel | String | qwen-max | 默认模型。新用户的默认模型编码 |
| DefaultThinkingMode | Int32 | 0 | 默认思考模式。Auto=0, Think=1, Fast=2 |
| DefaultContextRounds | Int32 | 10 | 上下文轮数 |
| MaxAttachmentSize | Int32 | 20 | 最大附件大小（MB） |
| MaxAttachmentCount | Int32 | 5 | 单次最多上传附件数 |
| AutoGenerateTitle | Boolean | true | 首条消息后是否自动生成会话标题 |
| TitlePrompt | String | 请用10个字以内… | 标题生成提示词模板 |
| AllowedExtensions | String | .jpg,.jpeg,… | 允许的文件扩展名 |
| EnableGateway | Boolean | true | 启用 API 网关 |
| GatewayRateLimit | Int32 | 60 | 网关限流。每分钟每用户最大请求次数 |
| UpstreamRetryCount | Int32 | 5 | 上游重试次数。模型返回 429 时最大重试 |
| EnableFunctionCalling | Boolean | true | 启用函数调用 |
| EnableMcp | Boolean | true | 启用 MCP 工具调用 |
| DefaultImageSize | String | 1024x1024 | 图像生成默认尺寸 |
| EnableUsageStats | Boolean | true | 启用用量统计 |
| BackgroundGeneration | Boolean | true | 后台继续生成 |

### 6.2 配置使用说明

- `ChatSetting.Current`：全局获取当前配置实例。
- 修改配置后自动保存到 `Config/ChatSetting.config`。
- 系统启动时自动加载，不存在则创建默认配置文件。

## 8. 标题自动生成策略

### 8.1 触发条件

1. 会话的 `MessageCount == 0`（首次发送消息时）。
2. 系统配置 `ChatSetting.AutoGenerateTitle == true`。
3. 会话标题为默认值"新建对话"。

### 8.2 实现流程

```
用户发送首条消息
    ├── 主流程：正常调用模型生成 AI 回复（SSE 流式）
    └── 异步任务：调用模型生成标题
         ├── 输入：用户首条消息内容
         ├── 提示词：ChatSetting.TitlePrompt + 消息内容
         ├── 使用模型：当前会话的模型（非流式调用）
         ├── 成功：更新会话标题
         └── 失败：保持默认标题，不影响主流程
```

### 8.3 前端交互

- 首条消息发送后，通过 SSE 事件或轮询获取会话标题更新。
- 建议在 `message_done` 事件中附加 `title` 字段（仅首条消息时返回）：

```
data: {"type":"message_done","usage":{...},"title":"关于量子计算的讨论"}
```

## 9. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| 1 | 不支持消息分支 | 简化前后端复杂度，编辑/重新生成直接替换 |
| 2 | 不做模型降级 | 模型不可用时直接报错，避免用户不知情使用非预期模型 |
| 3 | 复用魔方附件表 | 避免重复建设附件管理，魔方附件表已有完善的存储和管理能力 |
| 4 | 登录鉴权由外部模块提供 | 职责分离，NewLife.Cube 已有完整的用户和权限体系 |
| 5 | 共享有效期由后台配置 | 统一管理，避免每个用户自行设置导致管理困难 |
| 6 | SSE 区分思考/回答事件 | 前端可独立渲染思考过程折叠区域和正式回答 |
| 7 | 标题异步生成 | 不阻塞消息响应流程，用户体验更好 |
| 8 | 两层架构分离 | NewLife.AI（基础库）不依赖数据库，可单独使用；ChatAI（应用层）负责业务逻辑和持久化 |
| 9 | 不使用 Docker 沙箱执行代码 | 避免引入 Docker 依赖，降低部署复杂度 |
| 10 | 暂不含计费 | 先聚焦核心 Agent 能力，计费后续迭代；AppKey 用量统计已支持 |
| 11 | MCP 工具调用复用函数调用事件 | SSE 事件和前端组件统一，降低复杂度 |
| 12 | 后台继续生成 | 浏览器关闭不中断模型生成，用户重新打开可看到完整结果 |
| 13 | API 网关透明路由 | 按 model 字段自动匹配提供商，调用方无需关心具体后端 |
| 14 | AppKey 双维度统计 | 用量同时归属用户和 AppKey，支持双维度聚合查询 |
